# ClaudeDevKit

一个面向 Claude Code CLI 的轻量 AI 辅助开发工作流框架。

它的目标不是替代项目本身，而是为项目提供一层稳定、可维护、可追踪的 AI 协作机制，让 AI 能围绕路线图、当前目标、约束、测试和同步流程工作，而不是每次都从零猜项目结构。

## 适用场景

适合以下两类使用方式：

- 在一个已有项目中引入这套工作流框架
- 基于这套框架启动一个新项目，并让它从一开始就具备 AI 协作能力

这个框架强调：

- 轻量
- 开箱即用
- 命令少
- 文档驱动
- 可嵌入项目
- 可扩展

## 核心理念

这套框架只保留两份核心业务文档：

- `docs/ai/roadmap.md`
  项目总体技术设计与长期进度的唯一事实来源
- `docs/ai/current-goal.md`
  当前目标的执行文档，记录步骤、任务、阻塞、测试与同步说明

另外保留少量辅助状态文档：

- `docs/ai/current-goal.state.yaml`
- `docs/ai/project-tree.md`
- `docs/ai/project-summary.md`
- `docs/ai/change-log.md`
- `docs/ai/constraints/`

框架行为本身不写在大量说明文档里，而是尽量通过 Claude Code 原生命令、skills 和 agents 来承载：

- `.claude/commands/`
- `.claude/skills/`
- `.claude/agents/`

## 命令说明

| 命令 | 作用 | 是否只读 | 适用时机 |
|---|---|---|---|
| `/ai-help` | 显示命令说明、当前状态和推荐下一步 | 是 | 不清楚当前该做什么时 |
| `/ai-ask` | 只读问答，解释项目、代码、工作流 | 是 | 想问问题但不希望改任何内容时 |
| `/ai-init` | 初始化或修复工作流；空项目时可从技术路线图文档初始化 | 否 | 新项目启动或首次引入框架时 |
| `/ai-adopt` | 将本工作流安全接入一个正在开发中的现有项目 | 否 | 想把框架嵌入已有项目时 |
| `/ai-scan` | 扫描项目结构并刷新摘要信息 | 否 | 项目结构变化后或刚接入框架时 |
| `/ai-roadmap` | 维护长期路线图与总体技术设计 | 否 | 调整长期规划、插入目标、更新阶段时 |
| `/ai-goal` | 维护当前目标，从需求澄清到方案确认 | 否 | 准备开始一个具体目标时 |
| `/ai-check` | 检查当前目标状态、设计缺口、约束一致性 | 否 | 实现前、中、后做流程检查时 |
| `/ai-sync` | 将当前目标结果同步回 roadmap、状态与变更记录 | 否 | 当前目标完成并验证通过后 |
| `/ai-deadcode` | 检测高置信死代码并征询处置方式 | 否 | 需要清理冗余实现或收尾时 |
| `/ai-security` | 检测硬编码密钥、token 与明显安全问题 | 否 | 提交前、安全检查或同步前 |

## 核心工作流

标准开发流程如下：

1. 对于已在开发中的项目，先用 `/ai-adopt` 接入本工作流
2. 用 `/ai-help` 查看当前状态和下一步建议
3. 用 `/ai-roadmap` 维护总体技术方向和阶段目标
4. 用 `/ai-goal` 选定并设计当前目标
5. 方案确认后开始实现，并遵循 TDD
6. 用 `/ai-check` 做中间检查
7. 视情况运行 `/ai-deadcode` 和 `/ai-security`
8. 当前目标相关测试通过后，用 `/ai-sync` 回写路线图和状态
9. 只有在你确认后，才允许 commit 和 push

## 当前目标机制

框架要求任意时刻只有一个当前目标。

`current-goal` 的推进逻辑大致是：

- 检查当前工作区
- 若没有清晰目标，则与用户确认需求并给出方案选项
- 用户选择方案后，细化设计
- 把确认后的方案写入 `current-goal.md`
- 经用户确认后进入实现
- 实现过程遵循 TDD
- 测试通过后再同步回 `roadmap.md`

这样做的目的是让 AI 不会直接跳进代码，而是先和用户把当前目标说清楚。

## 路线图机制

`docs/ai/roadmap.md` 是整个框架的核心。

它同时承担两件事：

- 作为项目总体技术蓝图
- 作为长期路线和进度追踪表

其中的路线图表格至少包含：

- 目标ID
- 子目标ID
- 名称
- 描述
- 状态
- 前置依赖
- 风险/阻塞
- 验收结果
- 测试状态
- 实现时间
- Commit ID
- 备注

这意味着 AI 在规划、执行、同步时，都能围绕同一份主文档工作。

## 初始化方式

### 1. 引入到已有项目

把以下目录复制到目标项目中：

- `.claude/`
- `docs/ai/`

然后按顺序执行：

1. `/ai-adopt`
2. `/ai-roadmap`
3. `/ai-goal`
4. 进入实现前再用 `/ai-check` 做一次确认

最后根据项目实际情况补充：

- `docs/ai/constraints/project.md`
- `docs/ai/roadmap.md`

### 2. 作为新项目起点

如果你是在这套框架上新开一个项目，推荐顺序如下：

1. 先建立项目的最小骨架
2. 运行 `/ai-init`
3. 如果项目还很空，可以直接给 `/ai-init` 一份技术路线图文档
4. 让它初始化 `roadmap.md`、`current-goal.md` 等基础文档
5. 再用 `/ai-goal` 选定首个当前目标

注意：

- `/ai-init` 不负责直接生成业务代码
- 它只负责初始化文档、状态和候选目标

## 约束与边界

这套框架默认遵循以下原则：

- 不自动提交
- 不自动推送
- 不允许绕开用户确认直接进入实现
- 不允许创建无法嵌入现有项目的孤立代码
- 发现文档与代码不一致时，必须先询问用户如何处理

其中：

- 全局约束放在 `docs/ai/constraints/global.md`
- 项目专属约束放在 `docs/ai/constraints/project.md`

## 只读问答模式

如果你只想询问项目问题，而不希望 AI 修改任何内容，使用：

```text
/ai-ask
```

这个命令应当被视为严格只读：

- 可以读代码
- 可以读文档
- 可以解释问题
- 不可以改文件
- 不可以改状态
- 不可以触发同步、提交或推送

## 帮助入口

如果你不确定现在应该做什么，先运行：

```text
/ai-help
```

它应该告诉你：

- 当前有哪些命令
- 每个命令做什么
- 当前项目状态
- 现在最应该执行哪个命令

## 当前状态

这个仓库当前提供的是第一版工作流骨架，已经具备：

- 中文路线图模板
- 当前目标模板
- 命令定义
- skills 定义
- agents 定义

但它仍然更接近“可用模板”，不是已经过大量真实项目验证的成熟产品。最适合的使用方式是：

- 作为你自己的 AI 工作流起点
- 在真实项目中逐步迭代
- 根据语言、框架和团队习惯补充约束

## 后续建议

如果你准备继续完善这套框架，建议优先做这些事：

1. 增加一个技术路线图输入模板，专门给 `/ai-init` 使用
2. 为不同语言或框架补一组约束模板
3. 在真实项目中完整跑一遍 `roadmap -> goal -> implement -> sync`
4. 根据实际使用反馈压缩命令和文档内容
